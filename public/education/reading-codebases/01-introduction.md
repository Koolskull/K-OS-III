# Why most developers can't read other people's code

*This lesson is a stub. See `EDUCATION.md` for the project's curriculum and `docs/SEED_ISSUES.md` for the issue to flesh this out.*

Most developer learning fails not at syntax but at orientation. The skill of opening an unfamiliar repository — one with hundreds of files, conventions you didn't choose, history you weren't there for — and building a working mental model of it is undertaught. Bootcamps don't teach it. Most CS programs don't teach it. People mostly pick it up by suffering through it for years.

This track is a deliberate attempt to teach it directly. The lessons are short and tactical: how to spend the first thirty minutes in a new repo; how to use `git log`, `git blame`, and `git bisect` as learning tools; why reading the tests first is almost always the right move; how to build outward from entry points instead of trying to read the codebase top-to-bottom.
